üñ•Ô∏è Connect to Remote Systems
You need to remotely manage servers for security updates and monitoring. Learn how to connect securely using SSH and run commands on remote systems.
1
Basic SSH Connection
Connect to a remote server using SSH with username and password.
[admin@local ~]$ ssh user@192.168.1.100
Connect to server by IP address
[admin@local ~]$ ssh admin@webserver.company.com
Connect to server by hostname
2
Run Remote Commands
Execute commands on remote systems without opening an interactive shell.
[admin@local ~]$ ssh admin@webserver hostname
Get remote server's hostname
[admin@local ~]$ ssh admin@webserver "df -h"
Check disk space on remote server
3
Check Who's Connected
See who else is logged into the remote system for security monitoring.
[admin@webserver ~]$ w
Show who is logged in and what they're doing
[admin@webserver ~]$ who --ips
Show connected users with their IP addresses
4
Safely Disconnect
Properly close SSH connections when you're done.
[admin@webserver ~]$ exit
Close SSH session and return to local terminal
[admin@local ~]$ # Successfully disconnected from remote server
Back on your local system

üîë Set Up SSH Key Authentication
Passwords can be weak or stolen. Set up SSH key authentication for passwordless, more secure access to your servers. This is essential for automation and security.
1
Generate SSH Key Pair
Create a public/private key pair for secure authentication.
[admin@local ~]$ ssh-keygen -t rsa -b 4096
Generate strong 4096-bit RSA key pair
[admin@local ~]$ ssh-keygen -t ed25519
Generate modern Ed25519 key (recommended)
2
Copy Public Key to Server
Install your public key on the remote server for authentication.
[admin@local ~]$ ssh-copy-id admin@webserver
Copy your public key to remote server
[admin@local ~]$ ssh-copy-id -i ~/.ssh/id_ed25519.pub admin@192.168.1.100
Copy specific key file to server
3
Test Key Authentication
Verify that key-based authentication is working correctly.
[admin@local ~]$ ssh admin@webserver
Connect without being prompted for password
[admin@local ~]$ ssh admin@webserver whoami
Test remote command with key authentication
4
Protect Your Private Key
Secure your private key with proper permissions and passphrase.
[admin@local ~]$ chmod 600 ~/.ssh/id_ed25519
Ensure only you can read your private key
[admin@local ~]$ ls -la ~/.ssh/
Verify SSH directory permissions are secure
üí° What You Just Did
  • Generated secure SSH key pairs for passwordless authentication
  • Copied public keys to remote servers with "ssh-copy-id"
  • Tested key-based authentication to ensure it works properly
  • Secured private keys with proper file permissions

Create SSH Client Config

üîí Secure SSH Server Configuration
Your SSH server is exposed to the internet and under constant attack. Configure SSH security settings to protect against brute force attacks and unauthorized access.
1
Check Current SSH Configuration
Review current SSH server settings before making security changes.
[admin@server ~]$ sudo systemctl status sshd
Check SSH service status
[admin@server ~]$ sudo sshd -T | grep -E "^(port|permitrootlogin|passwordauthentication)"
Check current security settings
2
Disable Root Login
Prevent direct root access for better security.
[admin@server ~]$ sudo cp /etc/ssh/sshd_config /etc/ssh/sshd_config.backup
Backup SSH configuration before changes
[admin@server ~]$ sudo sed -i 's/#PermitRootLogin yes/PermitRootLogin no/' /etc/ssh/sshd_config
Disable root login via SSH
3
Enforce Key-Only Authentication
Disable password authentication to prevent brute force attacks.
[admin@server ~]$ sudo sed -i 's/#PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
Disable password authentication
[admin@server ~]$ echo "PubkeyAuthentication yes" | sudo tee -a /etc/ssh/sshd_config
Ensure public key authentication is enabled
4
Apply Security Configuration
Test and apply the SSH security configuration changes.
[admin@server ~]$ sudo sshd -t
Test SSH configuration for errors
[admin@server ~]$ sudo systemctl reload sshd
Apply changes without dropping connections
[admin@server ~]$ sudo systemctl status sshd
Verify SSH service reloaded successfully
üí° What You Just Did
  • Backed up SSH configuration before making security changes
  • Disabled root login to prevent direct admin access
  • Enforced key-only authentication to stop password attacks
  • Tested and applied configuration safely with "sshd -t" and "reload"
üîß Troubleshoot SSH Connection Problems
SSH connections are failing and you need to figure out why. Use debugging tools and log analysis to identify and fix SSH authentication and connection issues.
1
Use Verbose SSH Debugging
Enable detailed debugging to see what's happening during connection attempts.
[admin@local ~]$ ssh -v admin@webserver
Basic verbose output for debugging
[admin@local ~]$ ssh -vv admin@webserver
More detailed debugging information
2
Check SSH Key Problems
Verify SSH keys are properly configured and accessible.
[admin@local ~]$ ls -la ~/.ssh/
Check SSH directory contents and permissions
[admin@local ~]$ ssh-keygen -l -f ~/.ssh/id_ed25519.pub
Display public key fingerprint
3
Check Server SSH Logs
Look at server-side logs to understand why connections are failing.
[admin@server ~]$ sudo journalctl -u sshd -n 20
Check last 20 SSH service log entries
[admin@server ~]$ sudo tail -f /var/log/secure | grep sshd
Watch SSH logs in real-time
4
Fix Common SSH Problems
Resolve typical SSH authentication and permission issues.
[admin@local ~]$ ssh-keygen -R webserver
Remove old host key from known_hosts
[admin@local ~]$ chmod 700 ~/.ssh && chmod 600 ~/.ssh/id_ed25519 ~/.ssh/authorized_keys
Fix SSH file permissions

  • Used "ssh -v" and "ssh -vv" for detailed connection debugging
  • Checked SSH key files and their permissions
  • Examined server-side SSH logs for connection failures

ü§ñ Automate Tasks with SSH
You need to run the same security checks and updates across multiple servers. Create scripts that use SSH to automate repetitive administration and security tasks.
1
Create Server List
Maintain a list of servers for automated management.
[admin@local ~]$ cat > servers.txt << EOF\nwebserver\ndatabase\nfileserver\nmailserver\nEOF
Create list of servers to manage
[admin@local ~]$ cat servers.txt
Verify server list contents
2
Check Server Status
Create script to check basic status across all servers.
[admin@local ~]$ for server in $(cat servers.txt); do echo "=== $server ==="; ssh $server "hostname && uptime"; done
Check hostname and uptime on all servers
[admin@local ~]$ while read server; do echo "$server: $(ssh $server "free -h | grep Mem | awk '{print $3\"/\"$2}'")" ; done < servers.txt
Check memory usage on all servers
3
Run Security Updates
Automate security updates across multiple servers.
[admin@local ~]$ cat > security_update.sh << EOF\n#!/bin/bash\nfor server in $(cat servers.txt); do\n echo "Updating $server..."\n ssh $server "sudo dnf update -y --security"\n echo "$server update completed"\ndone\nEOF
Create security update automation script
[admin@local ~]$ chmod +x security_update.sh
Make the script executable
4
Collect Security Information
Gather security-relevant information from all servers.
[admin@local ~]$ for server in $(cat servers.txt); do echo "=== $server SSH users ==="; ssh $server "who"; done > security_report.txt
Collect logged-in users from all servers
[admin@local ~]$ parallel -j5 ssh {} "hostname && systemctl is-active firewalld sshd" ::: $(cat servers.txt)
Check security services in parallel (if parallel is installed)
